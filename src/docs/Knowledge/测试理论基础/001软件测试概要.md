# 软件测试定义

用一句话来概括软件测试的定义，则可以是**软件测试是一个为了发现软件错误(缺陷)而执行程序的过程**。

# 软件测试的意义

软件测试的核心意义在于，**通过软件测试为程序增加一些价值，即提高程序的可靠性或质量**。对于软件测试的定义存在着以下这些误解：

- 软件测试就是证**明软件不存在错误**的过程。
- 软件测试的目的在于证明**软件能够正确完成其预定的功能**。
- 软件测试就是建立一个“**软件做了其该做的**”信心的过程。

## “设计成功”的测试

从软件测试心理学上来讲，如果一开始就为了证明程序中不存在错误，则人总是会倾向于选择较少导致程序失效的测试数据。这会从主观上掩盖掉程序一些潜在的缺陷。**相对于程序开发是一个建设的过程，软件测试则更倾向于一个破坏的过程。**

一次“设计成功”的测试，应当能够发现程序中那些可以被修复的错误，或者可以最终确定再无其他可查出的错误。

## “设计失败”的测试

而设计失败的测试，则是指未能适当地对程序进行检查，导致未能找出错误。

# 软件测试的基本原则

- **测试不能证明软件没有缺陷，只能证明软件存在缺陷**；
- 完全的穷举测试是不可能的(巨量的输入输出组合)；
- 尽可能早地介入测试，利于在开发早期发现并修复缺陷，从而减少修复成本；
- 缺陷聚集：前期发现缺陷较多的部分应当加强测试；
- 杀虫剂悖论：重复同一测试用例，会导致新的缺陷无法被发现，因此测试用例需要定期评审与更新；
- **测试依赖于上下文**：测试策略根据项目背景、环境、风险、时间等进行相应调整
- 不存在缺陷的谬论：即使软件完全没有缺陷，如果它不符合用户需求和预期，依然是失败的。

# 软件测试数据

软件测试数据通常有以下来源：

- 生产环境数据：注意隐私和保密；
- 模拟数据
- 历史数据：可用于回归测试，需结合新生成的数据混合使用，确保测试覆盖率
- 第三方数据：从其他系统或服务商获取的数据，用于集成测试等场景；
- 随机数据
- 测试用例：有时测试用例本身就是具体的测试数据

# 软件测试结束的标准

- 测试用例执行完毕：计划内的测试用例全部执行，且结果符合预期；
- 漏洞修复完毕：所有**严重**和**高优先级**的漏洞已修复并通过回归测试；
- 软件质量达标：软件质量指标(BUG率、性能指标等)符合预期或已达到可接受的水平；
- 测试覆盖率：达到预定的测试覆盖率要求；
- 时间和资源：预分配的时间和资源已耗尽，进一步测试投入性价比不高；
- 风险评估：预估未发现的问题不会对产品的稳定性和业务功能产生重大影响；
- 管理审批：项目经理或产品主导方认为软件可以发布，给予正式批准；

# 产品上线标准

- 功能完备性：所有主要功能及关键业务流程完备可用
- 性能指标：高并发、高负载下的可用性
- 用户体验：人机交互原则
- 缺陷管理：高优先级和关键缺陷必须修复，回归后确认不再出现
- 安全性：没有漏洞和安全隐患
- 合法性
- 文档培训：完善的用户手册和技术文档以及人员培训
