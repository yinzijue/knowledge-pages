# 软件开发模型
## 瀑布模型
![[软件开发-瀑布模型]]

瀑布模型按工序将问题化简，将功能的实现与设计分开，便于分工协作。瀑布模型将软件生命周期划分为六个阶段，六个阶段自上而下相互衔接，次序固定。

**优点：**

+ 为项目提供了按阶段划分的检查点，用于质量把控；
+ 当前一阶段完成后，只需要专注于后续阶段即可；
+ 瀑布模型可以应用在其他迭代模型中；
+ 适用于较为固定的中小型项目；
+ 提供了一个模板，这个模板使得分析、设计、编码、测试、支持的方法可以在此有一个共同的指导。

**缺点：**

+ 过于理想，用户在缺乏实践的情况下，很难提出切合实际的需求；
+ 各阶段划分完全固定，阶段之间产生了大量文档(上一阶段的输出文档是下一阶段的输入文档)，极大地增加了工作量；
+ 由于开发模型是线性的，用户只能等到整个过程的末期才能看到开发成果，增加了开发风险；
+ 通过过多的强制完成日期和里程碑来跟踪各个项目阶段；
+ 无法评估项目进度，因为不知道哪个阶段会造成项目的延期；
+ 不能适应用户需求的变化。

## 演化模型
### 快速原型模型
![[软件开发-快速原型模型]]

快速原型模型在项目初期根据用户的需求快速构建一个可运行的系统原型，客户通过试运行它了解系统的全貌。后续用户会提出修改意见，研发人员据此修改原型...直至该原型满足用户的需求，后续就依此原型进行开发。

**优点：**

+ 克服了瀑布模型的缺点，减少了由于软件需求不明确带来的开发风险；
+ 适合预先不能确切定义需求的软件系统的开发；

**缺点：**

+ 所选用的开发技术和工具不一定符合主流的发展；
+ 快速建立起来的系统结构加上连续的修改可能会导致产品质量低下；
+ 产品原型在一定程度上可能会限制开发人员的创新。

### 螺旋模型
![[软件开发-螺旋模型]]

螺旋模型是在快速原型模型和瀑布模型的基础上，增加了风险分析策略，结合多种方法尽量降低风险，保证项目的产品质量。

**优点：**

+ 设计上的灵活性，可以在项目的各个阶段进行变更；
+ 以小的分段来构建大型系统，使成本计算变得简单容易；
+ 客户始终参与每个阶段的开发，保证了项目不偏离正确方向及项目的可控性；
+ 随着项目的推进，客户始终掌握项目的最新信息，从而能够和管理层有效交互；
+ 客户任何这种公司内部的开发方式带来的良好的沟通和高质量的产品；

**缺点：**

+ 建设周期较长，而软件技术发展较快，容易出现软件开发完毕后，和当前技术水平有了较大的差距，无法满足当前用户需求。
+ 很难让用户确信这种演化方法的结果是可控的。

# 软件测试模型
## V模型
软件测试V模型的另一个名字是软件开发过程中的RAD(Rapid Application Development,快速应用开发)模型。

![[软件测试-V模型]]

缺陷：测试被当作编码之后的一个阶段，忽略了测试对需求分析，系统设计的验证。

## W模型
![[软件测试-W模型]]

优点：

+ 测试与开发同步进行；
+ 测试的对象除了程序，还包括需求和设计；
+ 尽早发现软件缺陷可降低软件开发的成本；

缺陷：开发活动是串行的，测试和开发是线性的先后关系，上一阶段结束后才可以开始下一阶段的工作，不利于**迭代开发**。

## H模型
![[软件测试-H模型]]

优点：

+ 揭示了软件测试除了测试执行外，还有很多工作；
+ 测试活动完全独立，贯穿整个生命周期，且与其他流程并行；
+ 测试活动可以尽早准备、尽早执行，具有很强的灵活性；
+ 测试活动可视被测物的不同而分层次、分阶段、分次序执行，同时也可以被迭代；

缺点：

+ 管理型要求高：由于模型很灵活，必须要定义清晰的规则和管理制度，否则测试过程将非常难以管理和控制；
+ 技能要求高：H模型要求能够很好的定义每个迭代的规模，不能太大也不能太小；
+ 测试就绪点分析困难：测试很多时候，你并不知道测试准备到什么时候是合适的，就绪点在哪里，就绪点的标准是什么，这就对后续的测试执行的启动带来很大困难；
+ 对于整个项目组的人员要求非常高：在很好的规范制度下，大家都能高效的工作，否则容易混乱。例如：你分了一个小的迭代，但是因为人员技能不足，使得无法有效完成，那么整个项目就会受到很大的干扰。
