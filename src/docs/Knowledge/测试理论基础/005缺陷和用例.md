# 缺陷
## 定义
狭义概念上的缺陷是指软件的错误或漏洞。广义概念除此之外还包括测试工程师或用户所发现和提出的软件可改进的细节，或与需求文档存在差异的功能实现等。

## 记录原则
+ 缺陷编号：唯一标识符，便于追踪和管理；
+ 缺陷标题：简明扼要但准确描述缺陷；
+ 缺陷描述：问题描述、期望结果、实际结果；
+ 重现步骤
+ 环境信息：特定环境说明
+ 截图和日志文件
+ 优先级和严重性
+ 缺陷关联：注明缺陷与其他缺陷或任务之间的关联

## 产生原因
+ 需求阶段：需求描述不易理解，对需求的理解有歧义、错误等；
+ 设计阶段：设计文档存在错误或缺陷；
+ 编码阶段：代码出现错误；
+ 运行阶段：软硬件本身故障导致软件缺陷；
+ ...

## 类型划分
+ 软件未达到产品说明书标注的功能；
+ 软件出现了产品说明书指明不会出现的错误；
+ 软件功能超出了产品说明书指明的范围；
+ 软件功能未达到产品说明书虽未指出但应当达到的目标；
+ 软件测试人员认为软件难以理解、不易使用、运行速度缓慢，或者最终用户认为不好；

## 等级划分
+ Fatal:致命错误，造成系统崩溃、死机，或造成数据丢失、主要功能完全丧失等；
+ Critical:严重错误，指功能模块没有实现，主要功能部分丧失，次要功能全部丧失，或致命的错误声明；
+ Major:一般错误，次要功能模块丧失、提示信息不准确、用户界面差、操作时间长等；
+ Minor:轻微错误，一些小问题，例如个别错别字、文字排版不整齐等，对功能几乎没有影响，软件产品仍可以使用；

## 处理优先级
+ 立即修复，阻止进一步测试；
+ 在产品发布之前必须修复；
+ 如果时间允许，应该修复；
+ 可能会修复，但是也能发布；



## 管理流程和生命周期
## 管理流程
+ 报告BUG
+ 确认BUG
+ 修复BUG
+ 验证BUG修复
+ 关闭BUG

## 生命周期
+ Submitted: 已提交的缺陷
+ Asssigned：已指派
+ Open :确认“提交的缺陷”，等待处理
+ Rejected: 拒绝“提交的缺陷”，不需要修复或不是缺陷
+ Resolved :缺陷被修复
+ delayed：延期修复；
+ Closed :确认被修复的缺陷，将其关闭

## 不需要修复的缺陷
+ 没有足够的时间；
+ 不算真正的软件缺陷；
+ 修复风险太大；
+ 不值得修复；

# 用例
## 定义
测试用例是指对一项特定的软件产品进行**测试任务的描述**，体现测试方案、方法、技术和策略。简单来理解，可以认为测试用例是为了某个特殊目标而编制的一组测试输入、执行条件、以及预期的结果。

## 作用
+ **指导测试的实施；**
+ **规划测试数据的准备；**
+ **编写测试脚本的“设计规格说明书”；**
+ **评估测试结果的度量基准：**例如测试覆盖率、测试合格率、重要测试合格率；
+ **分析缺陷的标准：**通过收集缺陷，对比测试用例和缺陷数据库，分析是漏测还是缺陷复现；

## 组成部分
+ **<font style="color:#1DC0C9;">用例编号</font>**
+ 所属模块
+ **<font style="color:#1DC0C9;">用例标题</font>**
+ 用例优先级
+ **<font style="color:#1DC0C9;">前置条件</font>**
+ 测试数据
+ **<font style="color:#1DC0C9;">操作步骤</font>**
+ **<font style="color:#1DC0C9;">预期结果</font>**
+ **<font style="color:#1DC0C9;">执行结果</font>**
+ 用例状态
+ ...

## 设计步骤
+ 需求分析
+ 制订测试计划
+ 设计测试用例(输入数据、测试步骤、预期输出等)
+ 测试实现：编写自动化脚本、配置测试环境、准备测试数据
+ 测试执行
+ 测试评估

## 设计原则
+ 正确性：覆盖需求规格说明书各项功能，并且正常；
+ 全面性：覆盖所有的需求功能项；设计的用例除对测试点本身的测试外，还需考虑用户实际使用的情况、与其他部分关联的情况、非正常情况操作和环境设置等；
+ 连贯性：主次分明，单个用例不能同时覆盖很多功能点，否则执行起来牵连太大。
+ 可判断性：测试用例中要写清楚测试步骤，以及不同的操作步骤对应的测试结果；
+ 独立性：每个测试用例应当能够独立执行而不依赖其他测试用例的结果；
+ **<font style="color:#1DC0C9;">重用性和可维护性</font>**：易于理解和修改，可在类似测试场景中重复使用，因此避免过于复杂的描述；
+ 异常测试用例；
+ 自动化支持(可选的)
