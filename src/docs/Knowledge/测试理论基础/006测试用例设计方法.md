# 等价类划分法
## 基本信息
等价类划分法将程序所有可能的输入数据划分为若干等价类，然后从中选取具有代表性的数据当作测试用例进行合理的分类。测试用例由**有效等价类**和**无效等价类**的代表组成。等价类划分法可以提高测试的完备性和无冗余性。

## 优缺点

> [!note]+
>  + **优点**：等价类划分的测试用例设计方法减少了穷举法带来的大量测试用例，保证测试效果和测试效率，一般是有**输入性需求**的被测对象可以采用的方法；
> + **缺点**：输入与输入之间的关系考虑少，可能产生一些逻辑错误。还需要其他用例设计方法来补充测试。 



## 基本组件

> [!note]+
> + **有效等价类**：对于程序规格说明而言**合理的**、**有意义的**输入数据构成的集合，可以用来检验程序是否实现了规格说明预先规定的功能和性能；
> + **无效等价类**：对于程序规格说明而言不合理、无意义的输入数据构成的集合，可以用来找出程序异常说明情况，检查程序的功能和性能的实现是否有不符合规格说明要求的地方。

+ 弱一般等价类：遵循单缺陷原则，要求用例覆盖每一个变量的一种取值即可，取值为有效值；
+ 弱健壮等价类：在弱一般等价类的基础上，增加取值为无效值的情况；
+ 强一般等价类：遵循多缺陷原则，要求用例覆盖每个变量的每种取值之间的笛卡尔积，即**所有变量所有取值的所有组合**，取值为有效值。例如输入变量X有3个等价类，变量Y有2个等价类，则应设计3*2=6个用例覆盖所有有效等价类的组合；
+ 强健壮等价类：在强一般等价类基础上增加取值为无效值的情况。

## 等价类划分依据
+ 按区间划分；
+ 按数值划分；
+ 按数值集合划分；
+ 按限制条件或规划划分；
+ 按处理方式划分；
+ ...

## 等价类划分的原则
> [!note]+
>**在输入条件规定的取值范围或值下，可以确定一个有效等价类和两个无效等价类。**
例如：程序输入条件为10＜x＜100的数字，则有效等价类为10＜x＜100；2个无效等价类x≤10和x≥100。



> [!note]+
>**在规定了输入数据的一组n个数的值中，程序要对每个输入值分别处理的情况下，可以确定n个有效等价类和一个无效等价类。**

例如：程序输入x值取自于一个固定的枚举类型{1,3,7,15},且程序对该4个值分别进行处理，则4个有效等价类为枚举中的值；1个无效等价类为x≠1,3,7,15。



> [!note]+
>**在规定了输入数据必须遵守的规则或限制条件的情况下，可确定一个有效等价类（符合规则）和若干个无效等价类（从不同角度违反规则）**


例如：程序输入条件为奇数的整数x，则有效等价类为x的值为奇数的整数，其中一个无效等价类为x的值不为奇数的数。

> [!note]+
>**在确知已划分等价类中各个元素在程序中的处理方式不同的情况下，则应该再将该等价类进一步划分为更小的等价类，并建立等价类表。**

例如：程序输入条件为以‘a’开头，长度为8的字符串，并且字符串不包含‘a’-'z'之外的其他字符，则有效等价类为满足上述所有条件的字符串，无效等价类为不以'a'开头的字符串，长度不为8的字符串和包含了'a'-‘z’之外其他字符的字符串。

## 设计步骤
+ 确定等价类
+ 建立等价类表
+ 生成测试用例：编写测试用例，尽可能多地覆盖那些尚未被涵盖的有效等价类，直到所有有效等价类都被测试用例所覆盖；编写新的测试用例，每个测试用例只覆盖一个尚未被涵盖的无效等价类，直到所有无效等价类都被用例覆盖；

# 边界值分析法
## 基本信息
+ 边界值分析法是一种对**输入域**或**输出域**的边界值进行测试的一种黑盒测试方法。
+ 长期的测试工作经验告诉我们，大量的错误是发生在输入或输出范围的边界上，而不是输入、输出范围的内部，因此边界值分析法可以查出更多错误；
+ **边界值分析法常作为对等价类划分法的补充**，这种情况下，其测试用例来自于等价类的边界；

## 基本组件
边界值分析的基本思想是取输入边界的最小值、略高于最小值、正常值、略低于最大值和最大值作为测试点。考虑到健壮性测试，还可以加上略大于最大值和略小于最小值。它们可以分别被标记为min-，min,min+,nom,max-,max,max+。以下是在数轴上理解这些值。

![画板](https://cdn.nlark.com/yuque/0/2023/jpeg/23188736/1699082421102-5356e387-24ef-400c-b7b5-240480823c86.jpeg)

## 常见边界
通常情况下，软件测试所包含的边界值有几种类型：数字、字符、位置、重量、大小、速度、方位、尺寸、空间等。

相应的，需要注意的边界值有：最大/最小、首位/末位、上/下、最快/最慢、最高/最低、最短/最长、空/满等。

## 与等价类划分法的区别
+ 边界值分析不是从某个等价类中随便挑出一个作为代表，而是将这个等价类的每个边界都作为测试条件；
+ 边界值分析不仅考虑输入域，还要考虑输出空间产生的测试情况；
+ 等价类划分和边界值法都着重于考虑单个输入的输入条件；

# 因果图法
## 基本信息
+ 因果图法适用于有**多种输入条件组合，这些条件存在组合、约束关系，以及输入和输出条件之间存在因果关系**的测试场景。
+ **因果图法常与判定表法联合使用，**通过程序规格说明中的描述找出因(输入条件)和果(输出或程序状态的改变)，生成因果图，然后将因果图转换为判定表。

## 基本组件
### 因果
指的是输入条件和结果之间的关系。

![画板](https://cdn.nlark.com/yuque/0/2023/jpeg/23188736/1673601145482-802ce5c1-2ec6-4522-bbf9-2bf6b5298d8e.jpeg)

### 约束
约束是指**输入条件之间**还存在某种依赖关系。

![画板](https://cdn.nlark.com/yuque/0/2023/jpeg/23188736/1673602284628-b75fe2a7-e300-496e-84b1-d89f71b68a51.jpeg)

## 设计步骤
+ 根据需求说明或程序规格说明确定原因和结果；
+ 确定原因与原因之间的约束关系及原因与结果之间的因果关系，生成因果图；
+ 将因果图转化为判定表；
+ 根据判定表生成测试用例；
+ 补充健壮性测试用例；

# 判定表法
## 基本信息
+ 判定表法多用于有多个输入和多个输出，而且不同的输入之间有相互的组合关系、输入和输出之间有相互的制约和依赖关系。
+ 判定表法的使用有一些前提条件：
    - 规格说明以判定表形式给出，或很容易转换为判定表；
    - 条件的排列顺序不影响操作执行；
    - 规则的排列顺序不影响操作执行；
    - 当某一规则的条件已满足并确定要执行的操作后，不必校验别的规则；
    - 如果某一规则要执行多个操作，这些操作的执行顺序无关紧要；

## 基本组件
+ 条件桩：列出被测对象的所有输入，列出的输入条件与次序无关；
+ 动作桩：列出输入条件系统可能采取的操作，这些操作的排列顺序没有约束；
+ 条件项：列出输入条件的其他取值，在所有可能情况下的真假值；
+ 动作项：列出在条件项的各种取值情况下应采取的动作；
+ 规则：将条件项和动作项组合在一起，即在条件项的各种取值情况下应采取的动作。

在判定表中，贯穿条件项和动作项的每一列构成一条规则，即测试用例。可以针对每个合法的输入组合的规则设计测试用例进行测试，规则数量为2ⁿ个，其中n为输入条件的格式。

| 条件桩 | 条件项 |
| --- | --- |
| 动作桩 | 动作项 |


## 设计步骤
+ 明确条件桩；
+ 明确动作桩；
+ 对条件桩进行全组合；
+ 明确每个组合对应的动作桩；
+ 设计测试用例，每个规则对应一条测试用例；

## 方法案例
审核订购单：

+ 若金额大于500元，且订购单未过期，则发批准单和提货单；
+ 若金额大于500元，但订购单过期了，则不发批准单；
+ 若金额小于等于500元，则无论是否过期都发批准单和提货单，在过期的情况下还需发出通知单。



1. 明确条件桩：金额是否大于500，订购单是否过期；
2. 明确动作桩：发批准单；发提货单；发通知单；
3. 条件桩组合：金额大于500未过期；金额大于500过期；金额小于等于500未过期；金额小于等于500过期；
4. 明确规则：金额大于500未过期-发批准单+提货单；金额大于500过期-不发批准单；金额小于等于500未过期-发批准单+提货单；金额小于等于500过期-发批准单+提货单+通知单；
5. 设计测试用例：

| --- | --- | --- | --- | --- | --- |
| 条件桩 | 金额是否大于500 | Y | Y | N | N |
| | 是否过期 | N | Y | N | Y |
| 动作桩 | 发批准单 | Y | N | Y | Y |
| | 发提货单 | Y | | Y | Y |

# 正交试验设计法
## 基本信息
+ 正交试验法是研究多因素、多水平的一种实验法，它是利用正交表来对试验进行设计，通过少数的试验替代全面试验，**适用于组合量比较大的情况**。根据正交表的正交性从全面试验中挑选适量的、有代表性的点进行试验，这些有代表性的点具备了”均匀分散、整齐可比“的特点。
+ 正交表可分为**单一水平正交表**(各因素的水平数相同)和**混合水平正交表**(各因素的水平数不完全相同)

## 基本组件
+ n=试验次数，单水平计算公式为 n=c*(t-1)+1；混合水平计算公式为n=c1*(t1-1)+c2*(t2-1)...cx+(tx-1)+1其中，c为最多因素数量，t为各因素的水平数。**这种方法计算出的行数只是理论值，实际需要用已有的正交表去逼近这个理论值。**
+ t=水平数；
+ c=列数，即可能安排最多的因素个数；
+ L=正交表
+ 完整表达式：Ln(t^c),例如L4(2^3)表示需做4次实验，最多可观察3个因素，每个因素均为2个水平。
    - 混合水平正交表示例  L8(4^1 2^4),表示最多需要做8次实验，最多可观察5个因素，其中一个因素的水平数是4，另外四个因素的水平数为2；

| 实验次数\影响因素 | 1 | 2 | 3 |
| --- | --- | --- | --- |
| 1 | 1 | 1 | 1 |
| 2 | 1 | 2 | 2 |
| 3 | 2 | 1 | 2 |
| 4 | 2 | 2 | 1 |


## 设计步骤
+ 提取功能说明，构造**因子-状态**表；
+ 加权筛选，生成因素分析表：计算各因子和状态的权值，删去一部分权值较小，即重要性较小的因子或状态；
+ 构造正交表：利用逻辑命令来组合各因子的状态，做出布尔图；
+ 根据布尔图得到相应阶数的正交表；
+ 利用正交表每行数据构造测试用例；

## 选择合适的正交表
+ 考虑因素(变量)的个数；
+ 考虑因素水平(变量的取值)的个数；
+ 考虑正交表的行数
+ 取行数最少的一个；

选择正交表：

+ 表中的因素数≥3；
+ 表中至少有3个因素的水平数≥2；
+ 行数取最少的一个；
+ 选择正交表L4(2^3)；
+ 根据正交表每一行，生成测试用例；
+ 增补测试用例：不填写姓名、不填写身份证号、不填写手机号；

至此，按照各因素的每个水平数来考虑的话，需要2^3=8个测试用例，而通过正交试验设计，测试用例只需5个。

# 场景法
## 基本信息
+ 以上几种方法多侧重于单点功能方面，而场景法则更适用于解决业务比较复杂的系统或功能。场景法可以通过业务流将各个孤立的功能点串起来，为测试人员建立整体业务感觉，避免陷入功能细节而忽视业务流程要点的错误倾向。
+ **在拿到一个测试任务时，要遵循先整体后细节的原则，先关注主要业务流程和主要功能是否正确实现(场景法)，然后再对细节进行测试(等价类+边界值+判定表等)**

## 基本组件
### 基本流
基本流又叫有效流或正确流，用以模拟用户正确的业务操作流程。它具有以下特点：

+ 基本流只有一个起点，一个终点；
+ 基本流是主流；

### 备选流
备选流又叫无效流或错误流，用以模拟用户错误的业务操作流程。它具有以下特点：

+ 备选流是支流；
+ 备选流可以始于基本流，也可以始于其他备选流；
+ 备选流的终点，可以是一个流程的出口，也可以是回到基本流，还可以是汇入其他的备选流；
+ 备选流汇合时，谁汇入谁，取决于流量大小——即该流程出现问题的可能性大小，小的汇入大的；
+ 如果在流程图中出现了两个不相上下的基本流，一般需要把它们分别当作一个业务来看待；

+ 分析需求，确定业务流程(基本流、备选流)

+ 针对场景设计相应的测试用例；
