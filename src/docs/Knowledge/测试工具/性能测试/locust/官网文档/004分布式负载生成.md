# 分布式负载生成

单个运行Locust的进程可以模拟相当高的吞吐量。对于简单的测试计划和较小的数据负载，它每秒可以发出超过一千个请求，如果你使用FastHttpUser，则可能超过一万个请求。

但是，如果你的测试计划很复杂或者你想运行更多的负载，你需要扩展到多个进程，甚至可能是多台机器。幸运的是，Locust原生支持分布式运行。

为了实现这一点，你可以使用--master标志启动一个Locust实例，并使用--worker标志启动一个或多个实例。主实例运行Locust的Web界面，并告诉工作进程何时生成/停止用户。工作进程实例运行你的用户并将统计信息发送回主实例。主实例本身不运行任何用户。

为了简化启动过程，你可以使用--processes标志。它将启动一个主进程和指定数量的工作进程。它也可以与--worker一起使用，这时它只会启动工作进程。这个特性依赖于fork()函数，因此它不适用于Windows系统。

>[!note]+
由于Python无法在每个进程中充分利用多个核心（参见GIL），你需要为每个处理器核心运行一个工作实例，以便能够利用所有的计算能力。


>[!note]+
每个工作实例可以运行的用户数量几乎没有限制。只要它们的总请求率（RPS）不太高，Locust/gevent就可以在每个进程中很好地运行数千个甚至数万个用户。
如果Locust接近耗尽CPU资源，它会记录一个警告。如果没有警告但你仍然无法生成预期的负载，那么问题一定是其他的。

# 单台机器

启动一个主进程和4个工作进程非常简单：

```bash
$ locust --processes 4
```

  
你甚至可以自动检测你的机器上的逻辑核心数量，并为每个逻辑核心启动一个工作进程：

```bash
$ locust --processes -1
```

# 多台机器

在一台机器上以主模式启动locust：

```bash
$ locust -f my_locustfile.py --master
```

  
然后在每个工作机器上：

```bash
$ locust -f - --worker --master-host <你的主节点> --processes 4
```

注意

-f - 参数告诉Locust从主节点获取locustfile，而不是从它的本地文件系统。这个功能是在Locust 2.23.0版本中引入的。

# 多台机器，使用locust-swarm

当你对locustfile进行修改时，你需要重启所有的Locust进程。locust-swarm可以为你自动完成这个操作。它还解决了从工作节点到主节点的防火墙/网络访问问题，使用SSH隧道（如果主节点在你的工作站上运行，工作节点在数据中心中运行，这通常是一个问题）。

```bash
pip install locust-swarm
swarm -f my_locustfile.py --loadgen-list worker-server1,worker-server2 <任何其他常规的locust参数>
```

# 分布式负载生成的选项

--master-host <hostname or ip>  
可选地与 --worker 一起使用，以设置主节点的主机名/IP（默认为 localhost）。

--master-port <port number>  
可选地与 --worker 一起使用，以设置主节点的端口号（默认为 5557）。

--master-bind-host <ip>  
可选地与 --master 一起使用。确定主节点将绑定到哪个网络接口。默认为 *（所有可用接口）。

--master-bind-port <port number>  
可选地与 --master 一起使用。确定主节点将监听的网络端口。默认为 5557。

--expect-workers <worker 数量>  
当使用 --headless 启动主节点时使用。主节点将等待 X 个工作节点连接后才开始测试。

# 节点间通信

当在分布式模式下运行 Locust 时，您可能希望在主节点和工作节点之间进行通信以协调测试。这可以通过使用内置的消息钩子发送自定义消息来轻松实现：

``` python
from locust import events
from locust.runners import MasterRunner, WorkerRunner
# Fired when the worker receives a message of type 'test_users'
def setup_test_users(environment, msg, **kwargs):
    for user in msg.data:
        print(f"User {user['name']} received")
    environment.runner.send_message('acknowledge_users', f"Thanks for the {len(msg.data)} users!")
# Fired when the master receives a message of type 'acknowledge_users'
def on_acknowledge(msg, **kwargs):
    print(msg.data)
@events.init.add_listener
def on_locust_init(environment, **_kwargs):
    if not isinstance(environment.runner, MasterRunner):
        environment.runner.register_message('test_users', setup_test_users)
    if not isinstance(environment.runner, WorkerRunner):
        environment.runner.register_message('acknowledge_users', on_acknowledge)

@events.test_start.add_listener
def on_test_start(environment, **_kwargs):
    if not isinstance(environment.runner, WorkerRunner):
        users = [
            {"name": "User1"},
            {"name": "User2"},
            {"name": "User3"},
        ]
        environment.runner.send_message('test_users', users)
```

请注意，当在本地（即非分布式）运行时，此功能将被保留；消息将由发送它们的运行器直接处理。

>[!note]+
在注册消息处理程序时使用默认选项会以阻塞的方式运行监听函数，导致心跳和其他消息的执行时间被延迟。如果您认为您的消息处理程序需要运行超过一秒的时间，那么您可以将其注册为并发的。Locust 将会在新的 greenlet（协程）中运行它。请注意，这些 greenlet 永远不会被 join()。


更多详细信息，请参见完整示例。
